# Story 07A.2 — Redis container no Coolify

**Status:** Draft
**Phase:** 07A (Mastra core)
**Owner agent:** `@devops` (Operator)
**Story type:** Infrastructure
**Estimated effort:** 30-45 min

## Context

O design da Phase 07 escolhe **BullMQ + Redis** como pattern de processamento de mensagens (Decisão 4 aprovada pelo CEO: "opção C sem dúvidas, escala desde dia 1"). BullMQ depende de Redis como backend de filas. Phase 07A precisa de Redis disponível **antes** que a story 07A.4 implemente o package `packages/queue`.

Ambiente alvo: Coolify self-hosted em VPS (conforme `feedback_deploy_infra.md`). Dev local: Redis via Docker Compose ou serviço local.

## Acceptance Criteria

1. **Dev local:** Redis 7+ acessível em `redis://localhost:6379` via container Docker ou instância local
2. **Prod (Coolify):** Redis 7+ rodando como container dedicado com:
   - 512MB-1GB RAM reservado
   - Auto-restart em caso de crash
   - Volume persistente para dados (AOF ou RDB snapshots)
   - Health check nativo do Coolify
3. Variável `REDIS_URL` presente em:
   - `.env.example` (documentando o formato esperado)
   - `.env` local do @dev (valor real apontando pro dev local)
   - Variáveis de ambiente do Coolify em produção (valor real apontando pro container Redis interno)
4. **Teste funcional:** executar `redis-cli -u $REDIS_URL ping` retorna `PONG` em dev e prod
5. **Documentação:** seção nova em `docs/infrastructure/redis.md` com:
   - Como rodar localmente (comando Docker)
   - Como conectar (URL format)
   - Backup strategy (snapshot AOF)
   - Troubleshooting básico (connection refused, auth errors)
6. **Não expor** porta 6379 publicamente na VPS — Redis só acessível via rede interna do Coolify (security by default)
