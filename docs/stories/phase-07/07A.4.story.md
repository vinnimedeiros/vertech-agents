# Story 07A.4 — Packages de infraestrutura (queue + health)

**Status:** Draft
**Phase:** 07A (Mastra core)
**Owner agent:** `@dev` (Neo)
**Story type:** Infrastructure packages
**Estimated effort:** 60-90 min

## Context

Antes da Mastra core ser implementada, o projeto precisa de dois packages de infraestrutura horizontais:

1. **`packages/queue`** — wrapper de BullMQ + Redis pra processamento durable de jobs. Será usado por 07A.6 (worker de agent-invocation) e futuramente por Phase 08 (follow-up dispatcher, RAG ingest).
2. **`packages/health`** — contrato TypeScript uniforme e helpers pra criar endpoints de health check em qualquer package. Consumido por 07A.7 (endpoints) e por Phase 10c (Health Tech Dashboard consolidado).

Ambos são dependências transitivas da Mastra core (07A.5) e do fluxo runtime (07A.6).

## Acceptance Criteria

### Package `packages/queue`

1. Estrutura criada no workspace monorepo:
   - `packages/queue/package.json` com deps (`bullmq`, `ioredis`, `zod`)
   - `packages/queue/tsconfig.json` extends do tooling
   - `packages/queue/index.ts` exportando API pública
   - `packages/queue/src/config.ts` — conexão Redis a partir de `REDIS_URL`
   - `packages/queue/src/schemas.ts` — Zod schemas dos job payloads
   - `packages/queue/src/queues/agent-invocation.ts` — queue tipada
   - `packages/queue/src/telemetry.ts` — helpers pra expor métricas (consumido pelo health)
2. `config.ts` exporta:
   - `redisConnection` — `IORedis` singleton com reuso de connection
   - `defaultJobOptions` — objeto padrão: `{ attempts: 3, backoff: { type: 'exponential', delay: 2000 }, removeOnComplete: { count: 1000, age: 3600 }, removeOnFail: { count: 5000, age: 86400 } }`
3. `schemas.ts` exporta:
   - `agentInvocationJobSchema` (Zod): `{ messageId, conversationId, organizationId, scheduledFor? }`
   - `AgentInvocationJob` (inferred type)
4. `queues/agent-invocation.ts` exporta:
   - `agentInvocationQueue` — `new Queue<AgentInvocationJob>('agent-invocation', { connection: redisConnection, defaultJobOptions })`
   - `dispatchAgentInvocation(payload: AgentInvocationJob)` — helper pra enfileirar, implementando:
     - Validação Zod do payload
     - `jobId: payload.messageId` (idempotência — deduplicação em retries de webhook)
     - `deduplication: { id: \`conv:\${conversationId}\`, keepLastIfActive: true }` — **DIVERGÊNCIA APROVADA (Aria, 2026-04-19)**: serialização por conversa via feature nativa do BullMQ OSS, sem necessidade de BullMQ Pro nem lock Redis manual. Ver 07A.6.
5. `telemetry.ts` exporta:
   - `getQueueMetrics(queue: Queue)` → `{ waiting, active, completed_1h, failed_1h, dlq_count, oldest_job_age_seconds }`
6. `index.ts` re-exporta API limpa
7. Worker **NÃO** é implementado nesta story — fica pra 07A.6
8. Package adicionado no `package.json` raiz em workspaces (se já não estiver padrão de auto-discovery)

### Package `packages/health`

9. Estrutura criada:
   - `packages/health/package.json` com deps (`zod`)
   - `packages/health/index.ts`
   - `packages/health/src/contract.ts` — tipos `HealthStatus`, `HealthAlert`, `HealthCheckResult`
   - `packages/health/src/helpers.ts` — `defineHealthCheck()`, `formatMetrics()`, `aggregateStatus()`
   - `packages/health/src/components/database.ts` — checker Postgres
   - `packages/health/src/components/redis.ts` — checker Redis
10. `contract.ts`:
    ```typescript
    type HealthStatus = 'healthy' | 'degraded' | 'unhealthy';
    type HealthAlert = { severity: 'warning' | 'critical'; message: string; since?: string };
    type HealthCheckResult = {
      component: string;
      status: HealthStatus;
      metrics: Record<string, number | string>;
      alerts: HealthAlert[];
      timestamp: string; // ISO
    };
    ```
11. `helpers.ts`:
    - `defineHealthCheck<T>(component: string, checker: () => Promise<Omit<HealthCheckResult, 'timestamp' | 'component'>>) => () => Promise<HealthCheckResult>`
    - Preenche `timestamp` e `component` automaticamente
    - Wrap em try/catch — exceção vira status `unhealthy` com alert crítico
12. `components/database.ts`:
    - `checkDatabase()` → executa `SELECT 1`, mede latência, conta slow queries via `pg_stat_statements` (se disponível)
    - Status: `healthy` se latency < 100ms; `degraded` se 100-500ms; `unhealthy` se timeout ou erro
13. `components/redis.ts`:
    - `checkRedis()` → `PING`, `INFO memory`, `DBSIZE`, `CLIENT LIST | wc -l`
    - Retorna métricas + status
    - Reusa `redisConnection` do `packages/queue/src/config.ts` (import cruzado aceitável)

### Qualidade

14. `pnpm typecheck` passa
15. `pnpm build` (se o monorepo builda packages) passa
16. Unit tests básicos em Vitest (1 test por package é suficiente pra MVP): mocka Redis e DB, valida shape do output
17. README curto em cada package descrevendo uso esperado

## Notes

- **Boundary estrita:** `packages/queue` NÃO importa de `packages/ai`. `packages/health` NÃO importa de `packages/queue/src/queues/*` (só de `config.ts` pra conexão). Isso evita dependências circulares.
- **Idempotência:** `dispatchAgentInvocation` usa `jobId: messageId` (BullMQ dedupe por jobId). Se enfileirar duas vezes o mesmo `messageId`, o segundo é silenciosamente ignorado.
- Worker (processor) **NÃO** é iniciado aqui — só a queue. Worker vem na 07A.6.
- Se 07A.1 confirmar pattern de serialização via `groupKey` (BullMQ Pro), avaliar se vale migrar; caso contrário, usar lock Redis por conversationId na 07A.6.
