# Story 07A.6 — Runtime invoker + worker BullMQ + webhook dispatcher

**Status:** Draft
**Phase:** 07A (Mastra core)
**Owner agent:** `@dev` (Neo)
**Story type:** Feature — integration end-to-end
**Estimated effort:** 90-120 min

## Context

Conecta todas as peças construídas em 07A.3–07A.5:
- Schema `agent` + `messageStatusEnum` atualizados ✓
- `packages/queue` com fila tipada ✓
- `packages/ai` com Mastra + agente dinâmico ✓

Esta story entrega **o fluxo completo de uma mensagem de WhatsApp**:
webhook → persiste message QUEUED → enfileira job → worker puxa → invoker chama Mastra → resposta persistida → envia via WhatsApp → status SENT.

Ao fim desta story, **Vinni pode mandar mensagem do celular e o agente responde**. Validação via seed (07A.8).

## Acceptance Criteria

### Runtime invoker (`packages/ai/src/mastra/runtime/invoker.ts`)

1. Exporta `invokeAgentForMessage(params: { messageId: string })`:
   - Busca message + conversation + contact + agent do banco
   - **Guards** (abortam silenciosamente se não aplicável):
     - Message já está em `SENT` (idempotência de retry)
     - Conversation sem `isAIEnabled` ou sem `assignedAgentId`
     - Agent não está em status `ACTIVE`
   - Marca message `PROCESSING`
   - Monta `requestContext`: `{ agentId, organizationId, conversationId, contactId, whatsappInstanceId }`
   - Invoca `mastra.getAgent('commercial-agent').generate(message.text, { maxSteps: agent.maxSteps, requestContext, memory: { resource: 'ws-${orgId}:contact-${contactId}', thread: 'conv-${conversationId}' } })`
   - Recebe `response.text` + `response.usage` + `response.toolCalls` + `response.steps`
   - Persiste outbound message:
     - `senderType: 'AGENT'`, `senderId: agent.id`, `senderName: agent.name`, `senderAvatar: agent.avatarUrl`
     - `direction: 'OUTBOUND'`, `type: 'TEXT'`, `status: 'PENDING'`
     - `text: response.text`
     - `metadata: { toolCalls: response.toolCalls, usage: response.usage, steps: response.steps?.length, modelUsed, durationMs }`
   - Atualiza conversation: `lastMessageAt`, `lastMessagePreview` (primeiros 80 chars)
   - Se `conversation.channel === 'WHATSAPP'` e `conversation.channelInstanceId`:
     - Chama `sendText(channelInstanceId, contact.phone, response.text)` do `@repo/whatsapp`
     - Se sucesso → marca outbound message `SENT`
     - Se falha → marca `FAILED` com `metadata.error = String(err)`
2. **Tratamento de erro**: throw explícito se falha em persistir outbound (pra worker capturar e aplicar retry policy)
3. **Logging estruturado**: log no início com `messageId`, `agentId`, `modelUsed`; log no fim com duração e status

### Worker BullMQ (`packages/queue/src/workers/agent-invocation.ts`)

4. Cria `new Worker<AgentInvocationJob>('agent-invocation', processor, { connection, concurrency: 5 })`
5. Processor:
   - Valida payload via Zod schema
   - **Serialização por conversa**: antes de processar, adquire lock Redis `lock:conv:{conversationId}` com TTL 60s via `SET NX PX` (ou Redlock). Se não conseguir, re-enfileira com delay 1s.
   - Chama `invokeAgentForMessage({ messageId })` do `@repo/ai`
   - Libera lock ao fim (success ou failure)
6. Retry: já configurado via `defaultJobOptions` da 07A.4 (3 tentativas, backoff exponencial)
7. DLQ: jobs com 3 falhas vão pra DLQ automática do BullMQ (`agent-invocation:failed`) — inspecionáveis via Bull-Board (07A.7)
8. Graceful shutdown: handler de `SIGTERM` chama `worker.close()` e espera jobs em andamento
9. Script de execução: `pnpm worker:start` → roda `apps/worker/src/index.ts` ou similar que inicializa workers

### Webhook dispatcher (`packages/whatsapp`)

10. Implementar `emitConversationEvent(event, payload)` — atualmente stub em `packages/whatsapp`
11. Para evento `'message.inbound'`:
    - Valida payload
    - Chama `dispatchAgentInvocation({ messageId, conversationId, organizationId })` de `@repo/queue`
    - Se falhar ao enfileirar (Redis down), log erro crítico mas **NÃO** propaga (webhook já retornou 200 ao provider)
12. **Status QUEUED:** antes de chamar `emitConversationEvent`, o webhook handler (código existente do Phase 06) DEVE atualizar message `status = 'QUEUED'` ao invés de default `PENDING`. Ajuste mínimo no handler.

### Aplicação web

13. Dev mode: worker pode rodar no mesmo processo do Next.js (concurrency 5 já suficiente pro MVP)
    - Criar `apps/web/instrumentation.ts` que inicializa worker quando `NODE_ENV === 'development'` OU quando env `WORKER_MODE === 'inline'` (default em dev)
    - Em prod, worker roda em processo separado via `pnpm worker:start` no Coolify
14. `apps/worker/` novo app do monorepo OU script em `packages/queue/bin/worker.ts` — decidir pelo padrão mais enxuto do monorepo atual

### Qualidade

15. `pnpm typecheck` passa em todos os packages
16. Unit tests:
    - `invokeAgentForMessage`: mocka Mastra, banco, sender → verifica fluxo feliz + guards + error handling
    - Worker processor: mocka lock Redis → verifica serialização
17. Integration test (opcional, pode ficar pra 07A.8): enfileira job → espera worker processar → verifica outbound message persistida
18. `pnpm build` passa

## Notes

- **Lock Redis fallback:** se `SET NX PX` puro tiver race conditions, usar lib `redlock` (npm — multi-master lock). Decidir baseado em testes.
- **Serialização por conversa:** pattern escolhido evita agente responder fora de ordem se 2 mensagens chegam em < 1s. Trade-off: latência extra (~100ms de lock acquisition). Aceitável.
- **Idempotência:** `jobId = messageId` garante que re-enfileiramento do mesmo messageId é noop (BullMQ dedupe).
- **Status QUEUED vs PENDING:** convenção:
  - `PENDING` = mensagem criada mas não decidida (ex: draft de outbound por usuário humano)
  - `QUEUED` = esperando processamento no BullMQ
  - `PROCESSING` = worker pegou
  - `SENT` = enviada ao WhatsApp com sucesso
  - `FAILED` = deu ruim
