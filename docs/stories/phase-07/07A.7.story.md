# Story 07A.7 — Health endpoints API + Bull-Board

**Status:** Draft
**Phase:** 07A (Mastra core)
**Owner agent:** `@dev` (Neo)
**Story type:** Feature — observability
**Estimated effort:** 60-75 min

## Context

Conforme `feedback_health_tech_dashboard.md`, cada phase que cria infra instrumenta seu health endpoint. Phase 07A cria queue, Mastra, Redis — logo precisa expor os 4 endpoints iniciais. UI consolidada fica pra Phase 10c; aqui entregamos **endpoints JSON uniformes + Bull-Board em dev**.

Superadmin é o único role com acesso. Middleware Better-Auth já existente cobre o gate.

## Acceptance Criteria

### Endpoints API

1. `GET /api/admin/health/queue` em `apps/web/app/api/admin/health/queue/route.ts`:
   - Retorna `HealthCheckResult` conforme contrato de `@repo/health`
   - Métricas: `waiting`, `active`, `completed_1h`, `failed_1h`, `dlq_count`, `oldest_job_age_seconds`
   - Status: `degraded` se `waiting > 100` OU `oldest_job_age > 300`; `unhealthy` se Redis não responde
2. `GET /api/admin/health/mastra`:
   - Métricas: `active_conversations` (count em memória Mastra), `storage_connected`, `last_invocation_ago_seconds`
   - Status: `unhealthy` se storage desconectado; `degraded` se `last_invocation_ago > 3600` E `active_conversations > 0` (sinal de travamento)
3. `GET /api/admin/health/redis`:
   - Métricas: `connected`, `memory_used_mb`, `commands_per_second`, `connected_clients`, `maxmemory_policy`
   - Implementação usa `checkRedis()` do `@repo/health/components/redis`
4. `GET /api/admin/health/database`:
   - Métricas: `connected`, `avg_query_ms_5m`, `active_connections`, `slow_queries_1h`
   - Status: `degraded` se `avg_query_ms_5m > 500`
   - Usa `checkDatabase()` do `@repo/health/components/database`

### Gate de acesso

5. Todos os endpoints `/api/admin/**` passam por middleware que:
   - Valida sessão Better-Auth
   - Valida role `superadmin` (usar plugin admin do Better-Auth existente)
   - Retorna `401` se não autenticado, `403` se não superadmin
6. Middleware reaproveita lógica existente — não criar novo sistema de auth

### Bull-Board (dev only)

7. `GET /admin/queues` (fora de `/api`) monta Bull-Board UI
   - Biblioteca: `@bull-board/api` + `@bull-board/nextjs` ou equivalente compatível com Next.js 15 App Router (confirmado em 07A.1)
   - Acesso: mesma proteção superadmin
   - **Só ativa em `NODE_ENV === 'development'`** — em prod retorna 404

### UI pequena de índice (opcional, low-cost)

8. `GET /admin/health` em `apps/web/app/(admin)/admin/health/page.tsx`:
   - Server component que faz fetch paralelo dos 4 endpoints
   - Renderiza tabela simples: Componente | Status (badge colorida) | JSON expansível
   - Auto-refresh a cada 30s (client component wrapper com `useQuery` + `refetchInterval`)
   - **NÃO é o Health Tech Dashboard completo** (isso é Phase 10c) — é um placeholder funcional pra superadmin em 07A

### Qualidade

9. `pnpm typecheck` passa
10. Testar cada endpoint manualmente via `curl`:
    - Sem auth → 401
    - Com auth não-superadmin → 403
    - Com superadmin → 200 com JSON válido
11. Bull-Board acessível em dev
12. Nenhum endpoint expõe credenciais (REDIS_URL, DATABASE_URL, secrets de LLM) nas métricas
