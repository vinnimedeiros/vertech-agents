# Story 07B.2 — Shell do detalhe + header + menu lateral

**Status:** Ready
**Validation:** Vinni dispensou `@po *validate-story-draft` (pattern 07A). @dev pega direto após 07B.1 aprovada.
**Phase:** 07B (UI essencial)
**Owner agent:** `@dev` (Neo)
**Story type:** UI (layout)
**Estimated effort:** 75-90 min

## Context

Montar o esqueleto do detalhe do agente que as 6 abas (07B.3 a 07B.8) vão habitar: layout compartilhado com header, menu lateral vertical (pattern `SettingsMenu`) e área de conteúdo. Esta story **não** implementa nenhuma aba — entrega só o shell e a aba Identidade (07B.3) é quem de fato preenche a raiz `/agents/[agentId]`.

Esta é a story que define o pattern visual comum que as 6 abas replicam. Se o shell sai bom, as 6 abas entram rápido.

**Spec de referência:** `docs/phase-07/ui-spec-07b-agent-detail.md` — seções 3, 4 e 5 (shell, dirty state, pattern comum).

## Acceptance Criteria

### Layout (`layout.tsx` do detalhe)

1. Criar `apps/web/app/(saas)/app/(organizations)/[organizationSlug]/agents/[agentId]/layout.tsx`.
2. Layout busca o agente via `getAgentById(agentId)`. Se não existe → `notFound()`.
3. Valida que `agent.organizationId === activeOrganization.id` (cross-org attack guard). Se não bater → `notFound()`.
4. Renderiza:
   - `<AgentDetailHeader agent={agent} />`
   - Grid `grid-cols-1 lg:grid-cols-[240px_1fr] gap-6` com `<AgentSettingsMenu />` na esquerda e `{children}` na direita.
5. Se `agent.status === 'ARCHIVED'`: mostra banner no topo (acima do header):
   > "Este agente está arquivado. Desarquive pra editar." com botão `[Desarquivar]`
6. Agent é passado pro children via `AgentContext` (React context client-side) OU via prop drilling nas páginas filhas. **Decisão:** usar Context Provider client-side pra não re-fetch em cada aba.

### Header (`AgentDetailHeader.tsx`)

7. Componente client — precisa de inline edit do nome.
8. Layout row: `[Avatar 96px] [Nome editável + subtítulo] [Status badge] [Ações: Duplicar, Voltar pra lista]`.
9. Avatar: mostra `agent.avatarUrl` ou fallback com iniciais do nome.
10. Nome em `<h2>` ~28px bold. Click no nome transforma em `Input` inline; Enter salva via `updateAgentIdentity({ agentId, name })`, Esc descarta, blur salva se mudou.
11. Subtítulo abaixo do nome: `{role || 'Sem função definida'} • {modelo_label}`. Modelo_label vem de `SUPPORTED_MODELS.find(m => m.id === agent.model)?.label`.
12. Status badge à direita do nome. Click abre `DropdownMenu` com ações conforme status:
    - Se `DRAFT` ou `PAUSED`: "Ativar", "Arquivar"
    - Se `ACTIVE`: "Pausar", "Arquivar"
    - Se `ARCHIVED`: "Desarquivar" (único item)
13. Ativar pede validação: `name`, `role`, `model` preenchidos e `whatsappInstanceId` vinculado. Se faltar algo → toast com lista dos campos faltando, não ativa.
14. Arquivar → `AlertDialog`: "Tem certeza? O agente some da lista e é desvinculado do WhatsApp. Você pode desarquivar depois."
15. Ações secundárias (ícones à direita, `Button variant="ghost"`):
    - Duplicar (`CopyIcon`) → `duplicateAgent` → redireciona
    - Voltar pra lista (`ArrowLeftIcon`) → `Link` pra `/agents`

### Menu lateral (`AgentSettingsMenu.tsx`)

16. Client component — precisa de `usePathname` pra marcar ativo.
17. Estrutura: pode copiar/adaptar `SettingsMenu.tsx` ou herdar direto (OK escolher o que for mais limpo).
18. 6 itens fixos na ordem:

    | # | Label | Ícone | Path suffix |
    |---|---|---|---|
    | 1 | Identidade | `UserIcon` | `/` |
    | 2 | Persona | `SparklesIcon` | `/persona` |
    | 3 | Negócio | `BriefcaseIcon` | `/business` |
    | 4 | Conversas | `MessageSquareIcon` | `/conversation` |
    | 5 | Modelo | `CpuIcon` | `/model` |
    | 6 | WhatsApp | `PhoneIcon` | `/whatsapp` |

19. Item ativo: `border-l-2 border-primary font-bold text-foreground`. Inativo: `border-l-2 border-transparent text-foreground/70 hover:text-foreground`.
20. Mobile (`<lg`): vira lista horizontal rolável (`overflow-x-auto`) com borda inferior no item ativo (não esquerda). Herda pattern do `SettingsMenu`.
21. Cada item renderiza `<Link href={...}>` do `next/link`.

### Hook de form compartilhado (`use-agent-form.ts`)

22. Criar hook compartilhado pelas 6 abas:
    ```typescript
    function useAgentForm<T extends z.ZodType>({
      schema,
      defaultValues,
      action,           // server action específica da aba
    }): {
      form: UseFormReturn<z.infer<T>>;
      onSubmit: () => Promise<void>;
      isSubmitting: boolean;
    }
    ```
23. O hook encapsula:
    - `useForm` com `zodResolver(schema)`
    - Estado `isSubmitting` local
    - Função `onSubmit` que chama a action, toast de sucesso/erro, `reset(newValues, { keepDirty: false })` no sucesso
    - Navigation guard: se `isDirty` e user tenta navegar (`Link` click), `AlertDialog` "Sair sem salvar?" (usar `useNavigationGuard` custom ou `next-intl`-agnostic approach — pode ser feito com `beforeUnload` + intercept manual)

### Dirty state banner (`DirtyStateBanner.tsx`)

24. Client component que recebe `form` do react-hook-form e `onSubmit`/`onDiscard`.
25. Visível **apenas** se `form.formState.isDirty === true`.
26. Anatomia:
    - Bullet animado (`animate-pulse` primary)
    - Texto: "N mudança não salva" (singular) / "N mudanças não salvas" (plural), onde N = `Object.keys(form.formState.dirtyFields).length`
    - Botão `Descartar` (variant ghost) → chama `form.reset()`
    - Botão `Salvar` (variant primary) → chama `form.handleSubmit(onSubmit)()`. Disabled se `!isDirty || isSubmitting`. Spinner + "Salvando..." quando submitting.
27. Sticky: `sticky bottom-0 z-10 bg-card border-t-2 border-primary/20 px-6 py-3`.
28. i18n via `agents.detail.dirtyBanner` (spec seção 11).

### Deliverável visual da story

29. Após esta story, navegar pra `/agents/[agentId]` mostra:
    - Header completo (com inline edit do nome funcionando)
    - Menu lateral com 6 itens, Identidade marcada como ativa
    - Área de conteúdo com um placeholder `ComingSoon` "Aba Identidade em breve — story 07B.3" (será substituído em 07B.3)
    - Clicar em qualquer aba do menu (persona, business etc) leva a `notFound()` enquanto essas rotas não existem → aceitável. 07B.3-07B.8 preenchem.
30. Header responde: ativar/pausar/arquivar/desarquivar funcionam end-to-end. Inline edit do nome persiste no DB.
