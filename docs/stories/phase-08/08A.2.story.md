# Story 08A.2 — BullMQ worker ingest-document

**Status:** Ready for Review
**Phase:** 08-alpha
**Owner agent:** `@dev` (Neo)
**Story type:** Queue worker + background processing
**Estimated effort:** 2-3h

## Context

Conecta a função `ingestDocument` (entregue em 08A.1) ao sistema de queue existente (BullMQ + Redis, instalado em 07A). Quando upload endpoint (08A.4) enfileira um job, o worker puxa, chama a pipeline, e atualiza estado. Permite uploads pesados sem bloquear o chat do Arquiteto.

**Specs de referência:**
- Tech Spec § 5.2 (pipeline de ingest)
- Tech Spec § 9.3 (nova queue `ingest-document`)
- `packages/queue/src/` — código existente 07A de reuso

## Acceptance Criteria

### Definição da queue

1. GIVEN arquivo `packages/queue/src/queues/ingest-document.ts` criado, WHEN importado, THEN exporta `ingestDocumentQueue` com `name: 'ingest-document'` e defaults do BullMQ definidos em 07A (backoff exponencial 2s/8s/32s, attempts: 3, timeout: 120s).
2. GIVEN payload schema Zod em `packages/queue/src/schemas.ts`, WHEN validado, THEN inclui `{ documentId: string }` como único campo obrigatório (IDs já estão persistidos em knowledge_documents).

### Worker processor

3. GIVEN `packages/queue/src/workers/ingest-document.ts` criado, WHEN worker inicia, THEN consumes jobs da `ingest-document` queue.
4. GIVEN concorrência 5 configurada no worker (reusa pattern 07A), WHEN 5 jobs ativos simultaneamente, THEN processa em paralelo sem bloqueio.
5. GIVEN job fires, WHEN processor roda, THEN chama `ingestDocument(payload.documentId)` de `packages/ai/src/rag/ingest.ts`.
6. GIVEN `ingestDocument` completa com sucesso, WHEN job termina, THEN retorna `{ success: true, documentId, chunkCount }` (visível em QueueDash).
7. GIVEN `ingestDocument` lança erro, WHEN falha, THEN: job é marcado failed após 3 attempts, DLQ (Dead Letter Queue) guarda o job final com erro, evento `knowledge.document.failed` já foi emitido pela pipeline.
8. GIVEN documento com status ERROR persistido, WHEN job vai retentar, THEN pipeline `ingestDocument` detecta status e re-inicia (idempotente).

### Idempotência

9. GIVEN `jobId: 'ingest-' + documentId`, WHEN mesmo documento é enqueued 2x, THEN BullMQ dedupe automaticamente (não processa 2 vezes).
10. GIVEN documento já tem `status: READY`, WHEN job roda, THEN pipeline retorna early sem re-processar (otimização).

### Observabilidade

11. GIVEN worker processando, WHEN acessar QueueDash `http://localhost:3000/admin/queues` em dev, THEN aba `ingest-document` aparece com active/waiting/completed counts.
12. GIVEN health endpoint `/api/admin/health/queue`, WHEN GET, THEN métricas incluem dados da queue `ingest-document`.

### Integração com Realtime

13. GIVEN `ingestDocument` emite evento `knowledge.document.ready` via Supabase Realtime (canal `org:{orgId}:documents`), WHEN cliente está subscrito, THEN recebe evento ao vivo.
14. GIVEN erro permanente (após 3 retries), WHEN evento `knowledge.document.failed` emitido, THEN cliente pode mostrar estado de erro no chat do Arquiteto.

### Lifecycle

15. GIVEN app starts em prod, WHEN worker inicia, THEN conecta Redis, consome queue, loga "ingest-document worker ready".
16. GIVEN app shuts down gracefully (SIGTERM), WHEN worker para, THEN finaliza jobs ativos (ou marca pra retry) antes de desconectar.

### Completion Notes List

- Queue `ingest-document` + worker + dispatcher criados seguindo o **pattern 07A** da queue `agent-invocation`. Mesmos `defaultJobOptions` (3 attempts, backoff exponencial 2s/8s/32s), mesma lazy singleton com cleanup, mesma estrutura de worker com handlers de eventos.
- **Idempotência dupla (AC9 + AC10):** dispatcher usa `jobId = 'ingest-' + documentId` (BullMQ dedupe no enqueue), e a própria pipeline `ingestDocument()` retorna cedo em `status: READY` (já implementado em 08A.1).
- **Retry strategy (AC7, AC8):** o worker trata `result === null` como falha retornando throw, o que aciona o retry do BullMQ via `defaultJobOptions.attempts: 3`. Após exaurir tentativas, o job vai pro DLQ (contado pelo `getQueueMetrics.dlqCount`) e a row permanece `status: ERROR`. Se o job tentar de novo numa tentativa viva, a pipeline re-processa (nunca pula ERROR, só READY).
- **Realtime events (AC13, AC14):** a pipeline `ingestDocument` já persiste mudança de status em `knowledge_documents` (PROCESSING → READY|ERROR). Clientes subscrevem via Supabase Realtime no UPDATE da tabela — não precisa de emit explícito no worker. Story 08A.4 habilita o Realtime na tabela se ainda não está.
- **Health endpoint (AC11, AC12):** refatorei `/api/admin/health/queue` pra **agregar ambas as queues** via novo campo opcional `subchecks?: HealthCheckResult[]` no `HealthCheckResult`. Status do pai é a agregação dos filhos (pior status vence). Thresholds compartilhados (waiting > 100, oldest > 300s, dlq > 50). Cada queue aparece nomeada em `subchecks[].component`.
- **Bull-Board / QueueDash (parte do AC11):** a 07A.7 entregou endpoints de health mas não o setup do Bull-Board (`/admin/queues`). A queue `ingest-document` ficará automaticamente visível quando o Bull-Board for adicionado (Phase 10c ou quando necessário) — BullMQ expõe queues via Redis, basta o dashboard conectar.
- **Boot em 2 contextos (AC15, AC16):** `bin/worker.ts` (prod standalone) inicia ambos workers em sequência e faz graceful shutdown via `Promise.all([stopA, stopB])` em SIGTERM/SIGINT. `apps/web/instrumentation.ts` (dev inline) inicia ambos workers quando `NODE_ENV=development` ou `WORKER_MODE=inline`.
- **Desacoplamento de tipos:** apps/web não declara `bullmq` como dep direta. Re-exportei `type { Queue } from "bullmq"` no barrel do `@repo/queue` pra rota health tipar sem quebrar workspace boundaries.
- **Testabilidade do processor:** extraí a lógica pura pra `processIngestDocumentJob(data)` exportada, testada com `vi.mock("@repo/ai")`. 7 testes novos cobrem schema validation, idempotent shape, retry trigger, early-return behavior. Todos verdes (23/23 entre queue + health; 39/39 total dos pacotes tocados).
