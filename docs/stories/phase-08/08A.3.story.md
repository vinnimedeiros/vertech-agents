# Story 08A.3 — Registry architectTools (Mastra tools do Arquiteto)

**Status:** Ready for Review
**Phase:** 08-alpha
**Owner agent:** `@dev` (Neo)
**Story type:** Mastra tools registry + server logic
**Estimated effort:** 4-6h

## Context

Criar o registry `architectTools` com as 8 tools exclusivas do agente Arquiteto (ADR-001). Estas tools dão "mãos" ao Arquiteto pra gerar artefatos, refinar, aprovar, buscar conhecimento RAG, publicar agente (transação atômica) e evoluir agente pós-criação.

Cada tool tem Zod input/output completo, error cases tratados, e implementação do `execute`. Paridade 1:1 com configs da UI (Configurabilidade Tripla).

**Specs de referência:**
- Tech Spec § 4 (tool signatures completas): `docs/phase-09/tech-spec-arquiteto.md`
- ADR-001 (boundary): `docs/architecture/adr/adr-001-arquiteto-vs-orquestrador.md`
- Tech Spec § 6 (transação atômica de publicação, core do `publishAgentFromSession`)

## Acceptance Criteria

### Registry setup

1. GIVEN `packages/ai/src/mastra/tools/architect.ts` criado, WHEN importado, THEN exporta objeto `architectTools` com as 8 tools.
2. GIVEN `architectTools` consumido num `Agent` Mastra (story 09.5), WHEN Arquiteto invocado, THEN Mastra reconhece as tools pelas descriptions e schemas.

### Tool 1: `acknowledgeUpload`

3. GIVEN payload válido `{ documentId, brief_acknowledgment }`, WHEN `execute` roda, THEN busca knowledge_document com sessionId do requestContext, retorna `{ success: true, document: { id, fileName, status } }`.
4. GIVEN documentId não pertence à sessão ativa, WHEN execute roda, THEN retorna `{ success: false, error: 'DOCUMENT_NOT_FOUND' }`.

### Tool 2: `generateArtifact`

5. GIVEN tipo `business_profile|agent_blueprint|knowledge_base|final_summary`, WHEN execute, THEN lê working memory via Mastra, consolida campos da etapa em `content` estruturado (ver tech spec § 5.5), INSERTa em `agent_artifact`, atualiza `workingMemory.artifactIds`.
6. GIVEN checklist da etapa incompleto, WHEN tentar gerar, THEN retorna erro `CHECKLIST_INCOMPLETE` com lista de campos faltando.
7. GIVEN artefato gerado, WHEN retorna, THEN output inclui `{ success: true, artifact: { id, type, content, status: 'generated' } }`.

### Tool 3: `refineArtifact`

8. GIVEN `{ artifactId, instruction }`, WHEN execute, THEN busca artefato atual, chama LLM sub-call pra regenerar content aplicando instruction, UPDATEa row (`content` novo, `status: 'regenerated'`, `version + 1`), retorna `{ success, artifact, diff }`.
9. GIVEN artefato com `status: 'approved'`, WHEN tentar refinar, THEN retorna erro `ARTIFACT_LOCKED`.
10. GIVEN race condition (version column mudou), WHEN UPDATE executa com WHERE version = old, THEN falha, retorna erro `CONCURRENT_UPDATE`, Arquiteto pode retry.

### Tool 4: `approveArtifact`

11. GIVEN artifactId, WHEN execute, THEN UPDATEa `status: 'approved'`, `approvedAt: now()`, avança working memory (`currentStage` = próxima etapa, `checklist[stage].status = 'done'`), retorna `{ success, nextStage }`.
12. GIVEN artefato da etapa Criação aprovado, WHEN nextStage seria null, THEN retorna `nextStage: null` (Arquiteto sabe que é hora de publicar).

### Tool 5: `searchChunks`

13. GIVEN `{ query, topK=5 }`, WHEN execute, THEN chama `queryKnowledge(sessionId, query, topK)` de `packages/ai/src/rag/query.ts`, retorna `{ chunks: Array<{ content, similarity, documentId, documentTitle }> }`.
14. GIVEN retrieval sem hits, WHEN retorna, THEN `chunks: []` (não erro).

### Tool 6: `getDocumentKnowledge`

15. GIVEN nenhum input (usa sessionId do requestContext), WHEN execute, THEN retorna array de documentos da sessão com `{ id, title, status, summary, chunkCount }`.

### Tool 7: `publishAgentFromSession` (transação atômica)

16. GIVEN sessionId válida com status DRAFT, WHEN execute, THEN dentro de `db.transaction`: cria agent com dados consolidados do working memory, migra knowledge_documents (setter agentId, clear sessionId), atualiza metadata chunks, cria agent_version v1, marca session.status='PUBLISHED' + publishedAgentId, insere orchestrator_audit_log, emite eventos `agent.created` e `knowledge.migrated`. Tech spec § 6 define sequência exata.
17. GIVEN checklist incompleto (algum campo obrigatório null), WHEN execute, THEN retorna erro `CHECKLIST_INCOMPLETE` com lista, NÃO cria agent.
18. GIVEN falha em qualquer step, WHEN transação falha, THEN rollback automático Postgres, retorna erro estruturado, sessão permanece DRAFT.
19. GIVEN sucesso, WHEN retorna, THEN `{ success: true, agent: { id, name, status: 'DRAFT', version: 1 } }`.

### Tool 8: `updateAgentStructurally` (Chat de Evolução pós-criação)

20. GIVEN `{ agentId, changes, reason }`, WHEN execute, THEN dentro de `db.transaction`: aplica changes em agent, cria agent_version +1 snapshot, log em orchestrator_audit_log com `actorType: 'architect'`, emit evento `agent.updated`, retorna `{ success, newVersion, diff }`.
21. GIVEN agente em outra organização, WHEN tentar mexer, THEN retorna erro `FORBIDDEN` (validar via requestContext).

### Types + schemas

22. GIVEN output schemas Zod definidos para cada tool, WHEN Mastra type-check, THEN batem com o que execute retorna.
23. GIVEN errors estruturados (ex: `DOCUMENT_NOT_FOUND`, `CHECKLIST_INCOMPLETE`, `ARTIFACT_LOCKED`, `CONCURRENT_UPDATE`, `FORBIDDEN`), WHEN consumidos pelo LLM, THEN Arquiteto corrige e retenta (behavior via prompt).

### Completion Notes List

- **8 tools implementadas** seguindo assinatura da tech spec § 4 (Aria):
  1. `acknowledgeUpload` — valida doc pertence à sessão (AC3-4)
  2. `generateArtifact` — consolida WM em content, valida checklist, INSERT (AC5-7)
  3. `refineArtifact` — LLM sub-call (gpt-4o-mini) + optimistic locking via `version` column (AC8-10)
  4. `approveArtifact` — UPDATE status APPROVED, retorna nextStage (AC11-12)
  5. `searchChunks` — wrapper sobre `queryKnowledge` de 08A.1 (AC13-14)
  6. `getDocumentKnowledge` — lista docs da sessão com status/summary/chunkCount (AC15)
  7. `publishAgentFromSession` — transação atômica 10-step (AC16-19, tech spec § 6.1)
  8. `updateAgentStructurally` — Chat de Evolução pós-criação com tenant check (AC20-21)
- **Todas as tools** retornam shape discriminado por `success` (`true` ou `false`+`error code`). Errors estruturados via `ArchitectToolError` (AC23): `DOCUMENT_NOT_FOUND`, `SESSION_NOT_FOUND`, `CHECKLIST_INCOMPLETE`, `ARTIFACT_NOT_FOUND`, `ARTIFACT_LOCKED`, `CONCURRENT_UPDATE`, `AGENT_NOT_FOUND`, `FORBIDDEN`, `INVALID_ARTIFACT_TYPE`, `MISSING_CONTEXT`, `PUBLISH_FAILED`, `NOT_IMPLEMENTED_YET`.
- **Working memory contract:** tech spec § 3.1 define o shape completo. A story 09.5 (Agent Arquiteto Mastra) vai persistir via `@mastra/memory` Zod schema. Nesta phase, as tools **consomem working memory via `runtimeContext.get('workingMemory')`** — guard `requireArchitectContext` falha cedo (`MISSING_CONTEXT`) se o Agent não populou. Documentado no header do helpers.ts. Commitment pro 09.5: ao montar Agent Arquiteto, popular esses 4 keys no requestContext da invocação.
- **Transação atômica (publishAgentFromSession, tech spec § 6):** 10 steps exatos, mas **ordem otimizada** — validação de checklist acontece FORA da tx (early fail sem abrir conexão). Dentro da tx: session guard → INSERT agent → migrate knowledge_documents → UPDATE chunks metadata via `jsonb_set` chained (agentId + null sessionId) → INSERT agent_version v1 → UPDATE session PUBLISHED → INSERT audit log. Events do step 10 via Realtime Supabase implícito (cliente subscrevê UPDATE das tabelas).
- **Optimistic locking (refineArtifact, AC10):** `UPDATE agent_artifact SET ... WHERE id = X AND version = old.version`. Zero rows afetadas → throw `CONCURRENT_UPDATE` pro Arquiteto retentar. Pattern pesquisado em Drizzle, valida via `.returning()` length.
- **LLM sub-call (refineArtifact):** `gpt-4o-mini` via `generateObject` do `ai` v4 com schema `z.record(string, any)` — LLM precisa retornar JSON no shape do input. Se shape quebrar, Zod parse falha e cai no catch externo. Prompt pt-BR estruturado: contexto + tipo + instrução + JSON atual.
- **Tenant isolation (updateAgentStructurally, AC21):** busca agent, compara `existing.organizationId !== context.organizationId` → `FORBIDDEN`. Todas as outras tools naturalmente escopadas por `sessionId` pertencente ao user.
- **Derivação de tools das capabilities** no publish: pattern `CAPABILITY_TO_TOOLS` map (qualification → [updateLeadData, moveLeadStage], scheduling → [scheduleMeeting], faq → [searchKnowledgeBase], handoff → [handoffToHuman], followup → [createLeadActivity]). Dedupe manual via hash map (evita iteração de Set pra manter compat com ES5 target do web).
- **Zod dep adicionada** em `@repo/ai/package.json` (^3.25.76 pra satisfazer peer de @mastra/core v1.25.0). `pnpm install` ok.
- **29 testes novos** cobrem helpers, validações, error classes, context guards. Rollback real da transação Postgres fica pra 08A.5 (DB live + fixtures).
