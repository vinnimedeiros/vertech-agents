# Story 08A.4 — Upload endpoint + Supabase Storage bucket

**Status:** Ready for Review
**Phase:** 08-alpha
**Owner agent:** `@dev` (Neo)
**Story type:** API route + Storage config
**Estimated effort:** 2-3h

## Context

Endpoint HTTP `/api/architect/upload` que recebe arquivo(s) do usuário durante conversa com o Arquiteto, salva no Supabase Storage, cria row em `knowledge_document` com status PENDING, e enfileira job na queue `ingest-document` (08A.2) pra processar em background.

Também estabelece o bucket `architect-uploads` no Supabase Storage com policies apropriadas.

**Specs de referência:**
- UI Spec § 7 (Composer com upload inline): `docs/phase-09/ui-spec-arquiteto.md`
- Tech Spec § 1.2 e § 9.2 (endpoints): `docs/phase-09/tech-spec-arquiteto.md`

## Acceptance Criteria

### Setup do Storage bucket

1. GIVEN Supabase project `agents-v2`, WHEN bucket `architect-uploads` criado (via MCP ou dashboard), THEN é `public: false`, com size limit 10MB por arquivo.
2. GIVEN bucket criado, WHEN policy aplicada via SQL/MCP, THEN: SELECT/INSERT permitido pra membro da org (via JOIN em member table), DELETE só por admin da org.
3. GIVEN bucket estruturado, WHEN arquivo enviado, THEN path segue pattern `{organizationId}/{sessionId-or-agentId}/{documentId}/{filename}`.

### Endpoint POST /api/architect/upload

4. GIVEN route `apps/web/app/api/architect/upload/route.ts` criado, WHEN POST multipart/form-data, THEN aceita campos: `sessionId` (string, uuid), `file` (File único ou múltiplo).
5. GIVEN auth middleware, WHEN request sem session, THEN retorna 401.
6. GIVEN session válida + `sessionId` pertencente ao user atual, WHEN autoriza, THEN segue; senão 403.
7. GIVEN file tipo não suportado, WHEN valida, THEN retorna 400 com erro `INVALID_FILE_TYPE` + lista dos suportados (pdf/docx/csv/xlsx/txt + png/jpg/webp pra imagem).
8. GIVEN file >10MB, WHEN valida, THEN retorna 400 com `FILE_TOO_LARGE`.
9. GIVEN 6+ anexos simultâneos na mesma mensagem, WHEN valida, THEN retorna 400 com `TOO_MANY_ATTACHMENTS`.

### Upload para Storage

10. GIVEN arquivo válido, WHEN upload, THEN: cria documentId via cuid, sobe pro Storage com path pattern, insere row em `knowledge_document` com `status: 'PENDING', title: fileName, fileType: MIME mapeado, fileSize`, enfileira job `ingest-document` com `{ documentId }`.
11. GIVEN múltiplos arquivos (até 5), WHEN upload, THEN processa em paralelo, retorna array com resultados por arquivo.
12. GIVEN sucesso, WHEN retorna, THEN body: `{ documents: Array<{ id, fileName, fileType, fileSize, status: 'PENDING' }> }`.

### Endpoint POST /api/architect/upload-link (anexo de URL)

13. GIVEN route com POST `{ sessionId, url }`, WHEN URL válida, THEN: cria documentId, title = HTML title da página (fetch), insere knowledge_document com `fileType: 'URL'`, `fileUrl: url`, enfileira job.
14. GIVEN URL inválida (não-http(s) ou timeout), WHEN valida, THEN 400 com erro claro.

### Error handling

15. GIVEN upload falha no Storage, WHEN handler captura, THEN NÃO cria row em knowledge_document, retorna 500 com erro.
16. GIVEN row criada mas enqueue falha, WHEN detectado, THEN marca `knowledge_document.status: 'ERROR'`, retorna 500 com orientação.

### Rate limit do endpoint

17. GIVEN user fez 10 uploads em 60s, WHEN 11º request, THEN 429 com `RETRY_AFTER`.

### Completion Notes List

- **Bucket `architect-uploads` criado em prod** via MCP Supabase (`apply_migration`). Config: private, 10MB limit, 8 MIME types (pdf, docx, csv, xlsx, txt, png, jpg, webp). Verificado via SQL: `public=false, file_size_limit=10485760, mime_count=8`.
- **Storage RLS policies:** default deny pra roles `anon` e `authenticated` escopado ao bucket. O projeto usa better-auth (não Supabase Auth), então policies complexas com `auth.uid()` não funcionam. service_role ignora RLS e é usado pelo endpoint server-side. Documentado no SQL migration + upload-helpers.ts.
- **2 endpoints criados:**
  - `POST /api/architect/upload` — multipart, processa até 5 files em paralelo via `Promise.allSettled` (um arquivo falhar não derruba o lote). Retorna `{ documents: [...], errors: [...] }`.
  - `POST /api/architect/upload-link` — JSON body, valida http/https, fetch com timeout 10s (AbortController) + cap de 2MB HTML, extrai `<title>` via cheerio + fallback `og:title` + fallback hostname.
- **Divergência documentada (AC7):** imagens `png/jpg/webp` **não** são aceitas pelo endpoint apesar do bucket permitir. Razão: não há extractor em 08A.1 pra imagens, e o enum `knowledge_document.fileType` não tem `IMAGE`. Aceitar sem processar criaria docs fantasmas em PENDING. A AttachmentMenu (09.4) pode ter fluxo separado pra preview de imagens se necessário. Ajuste minimal na spec.
- **Rate limit** — implementei Redis-based usando a conexão singleton do `@repo/queue`. Pattern: `INCR` + `EXPIRE` (first hit), pragmático pra user-level com tradeoffs aceitáveis em race extremo. Config exportada: `ARCHITECT_UPLOAD_LIMIT` (10 req / 60s).
- **Path building (AC3):** `{orgId}/{sessionId}/{docId}/{safeFilename}`. Sanitização do filename: NFD normalize + strip diacriticos via `\p{Diacritic}` Unicode property + substitute de chars inválidos por `_` + truncamento 80 chars + fallback `file` se só não-alfanumérico. Extensão mantida separada e validada.
- **Insert+enqueue idempotente:** gero o `documentId` via `createId` de cuid2 ANTES do upload (pra path determinístico) e passo explícito ao insert. Re-exportei `createId` do `@paralleldrive/cuid2` no `@repo/database` barrel pra não precisar adicionar dep direta em apps/web.
- **Handling robusto (AC15-16):**
  - Se Storage falha → `UploadError("STORAGE_FAILED", 500)`, row NÃO é criada
  - Se insert sucede mas enqueue falha → UPDATE row pra `status: ERROR` + errorMessage, retorna `ENQUEUE_FAILED` (AC16 exato)
- **Cheerio** adicionado como dep em `apps/web/package.json` (v1.2.0).
- **24 testes novos** cobrem: MIME mapping (5 tipos + imagens + desconhecidos), validação (tipo/tamanho/vazio), path building (diacriticos, separators, truncamento, fallback), UploadError shape.
