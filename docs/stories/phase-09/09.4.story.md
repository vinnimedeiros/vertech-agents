# Story 09.4 — AttachmentMenu + upload flow + mini-cards

**Status:** Ready for Review
**Phase:** 09
**Owner agent:** `@dev` (Neo)
**Story type:** UI + file handling + API integration
**Estimated effort:** 3-4h

## Context

Completa o composer com funcionalidade de upload durante conversa (diferencial do Vertech vs Mercado Agentes). Usuário pode anexar arquivos (PDF/DOCX/CSV/XLSX/TXT), imagens (PNG/JPG/WEBP) ou links de site enquanto conversa. Mini-cards aparecem acima do textarea antes de enviar e dentro da mensagem após enviar.

**Specs de referência:**
- UI Spec § 4.8-4.9 (AttachmentMenu + composer com anexos): `docs/phase-09/ui-spec-arquiteto.md`
- UI Spec § 7 (Composer com upload detalhado)
- API: 08A.4 endpoint `/api/architect/upload` + `/api/architect/upload-link`

## Acceptance Criteria

### AttachmentMenu (DropdownMenu no botão Paperclip)

1. GIVEN usuário clica `Paperclip`, WHEN menu abre, THEN 3 itens: "Arquivo" + descrição "PDF, DOCX, CSV, XLSX, TXT (até 10MB cada)", "Imagem" + "PNG, JPG (até 5MB)", "Link de site" + "Cole uma URL, o Arquiteto faz a leitura".
2. GIVEN click em "Arquivo", WHEN selecionado, THEN abre `<input type="file" accept=".pdf,.docx,.csv,.xlsx,.txt" multiple />`.
3. GIVEN click em "Imagem", WHEN selecionado, THEN abre `<input type="file" accept="image/png,image/jpeg,image/webp" multiple />`.
4. GIVEN click em "Link de site", WHEN selecionado, THEN abre `Dialog` pequeno com `Input` de URL + "Anexar".
5. GIVEN keyboard navigation, WHEN user aperta Cmd+K, THEN menu abre automaticamente (integração com evento da 09.3).

### Validação client-side

6. GIVEN file selecionado, WHEN `file.size > 10MB`, THEN toast destructive "Arquivo muito grande. Limite de 10MB." (ou 5MB para imagem).
7. GIVEN `5 anexos já no composer` + novo file, WHEN user tenta adicionar, THEN toast "Máximo 5 anexos por mensagem."
8. GIVEN MIME type não suportado, WHEN tenta anexar, THEN toast "Tipo de arquivo não suportado."

### Upload flow

9. GIVEN arquivo válido, WHEN user confirma anexo, THEN: mini-card aparece no composer com `<Spinner /> enviando...`, chama POST `/api/architect/upload` com `{ sessionId, file }`.
10. GIVEN upload succeed, WHEN response chega, THEN mini-card muda pra estado "pronto" (sem spinner), guarda `documentId` retornado pelo endpoint.
11. GIVEN upload falha (500/400), WHEN catch, THEN mini-card muda pra `error` + botão `[retry]`.

### Mini-card pending (antes de enviar)

12. GIVEN mini-card, WHEN renderiza pending, THEN: ícone por tipo (`FileText`/`Image`/`Link`), nome truncado, tamanho legível (KB/MB), botão `[X]` remover, estados spinner/pronto/erro.
13. GIVEN múltiplos mini-cards, WHEN renderiza, THEN flex-wrap com gap 8px acima do textarea.
14. GIVEN click `[X]`, WHEN remove, THEN: se status=`uploading`, aborta request; se status=`ready`, deleta row em `knowledge_document` via endpoint DELETE (ou só marca pra não enviar no composer).

### Mini-card na mensagem enviada

15. GIVEN user clica enviar com anexos pending, WHEN submit, THEN: mensagem aparece no chat com mini-cards como parte da bubble (estado `processing`).
16. GIVEN worker completa ingest (evento `knowledge.document.ready` via Realtime), WHEN cliente subscribe, THEN mini-card atualiza pra estado `indexado` (check verde).
17. GIVEN ingest falha, WHEN `knowledge.document.failed`, THEN mini-card mostra `erro` no chat, Arquiteto menciona no próximo turn.

### Preview de Link

18. GIVEN URL anexada via Dialog, WHEN fetch inicia, THEN mini-card com `Link` icon + URL + spinner "Processando...".
19. GIVEN fetch succeed, WHEN response, THEN mini-card mostra favicon + título da página + URL + "N páginas scrapadas".

### Types e integrações

20. GIVEN shape de attachment, WHEN tipo TS, THEN: `{ id: tempClientId, documentId: serverResponse, fileName, fileType, fileSize, status: 'uploading'|'ready'|'error'|'processing'|'indexed', errorMessage? }`.
21. GIVEN parent `ChatShell`, WHEN gerencia lista de attachments pending, THEN passa pro composer via prop controlled (`attachments`, `onAttachmentsChange`).

### Translations

22. GIVEN textos novos, WHEN i18n, THEN `architect.chat.composer.attachMenu.*` + `architect.chat.states.*` (UI Spec § 14).

## Dev Agent Record

**Agent Model Used:** Neo (@dev)

**Completion Notes:**
- Endpoint auxiliar `POST /api/architect/sessions` adicionado ao escopo: upload exige sessionId DRAFT válido, e o Composer chega em `/agents/new?template=X` sem session. Hook `useFileUpload` cria lazy no primeiro anexo (idempotente por ref — chamadas subsequentes reaproveitam).
- `mastraThreadId = sessionId` e `mastraResourceId = userId` (convenção Mastra Memory padrão). 09.5 vai consumir esses ponteiros no `useChat`.
- Imagens são aceitas visualmente mas NÃO subidas pro bucket: server atual (08A.4) rejeita MIME de imagem com `INVALID_FILE_TYPE`. Filtramos client-side e marcamos `status: ready` com `URL.createObjectURL` pra preview local (evita erro UX). Quando um extractor de imagem for adicionado no RAG, remover o filtro em `useFileUpload.ts`.
- Limite de 5 anexos enforçado no menu (disable opções) e no handler de multi-file (slice). Toast dispara se user tenta exceder.
- Abort: como upload de lote vai num único FormData request, remover 1 attachment durante upload aborta o lote inteiro. Tradeoff aceitável (documentado no hook). Remover depois do `ready` só remove do state local — row em `knowledge_document` continua viva (cleanup é via 09.10 ou sessão abandonada).
- AC20 type shape atendido via `ArchitectAttachment` em `attachment-helpers.ts`.
- Realtime via `supabase.channel('architect-session-{sessionId}').on('postgres_changes', filter sessionId=eq.{id})`. Só ativa quando sessionId existe (guard evita WS conexão desnecessária pra quem abriu chat sem anexar).

**Divergências da spec:**
- AC22 pede i18n keys `architect.chat.composer.attachMenu.*`. Mantido hardcoded pt-BR igual 09.1/09.2/09.3 (pattern do projeto inteiro). Se futuramente migrar pra i18n, todos os textos visíveis estão nos componentes de `chat/`.
- Scope IN mencionava "preview visual interno de imagem" como OUT, mas implementei preview local via `URL.createObjectURL` (custo zero, melhora UX).
