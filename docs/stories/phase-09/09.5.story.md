# Story 09.5 — Mastra Architect Agent + instructions template + route handler

**Status:** Ready for Review
**Phase:** 09
**Owner agent:** `@dev` (Neo)
**Story type:** Mastra agent + backend + frontend integration
**Estimated effort:** 6-8h (maior story da Phase 09)

## Context

O coração da Phase 09. Cria o agente Mastra Arquiteto com memory config completa (working memory estruturada via Zod), instructions template dinâmico por vertical, route handler `/api/architect/chat` com streaming SSE, rate limit via Redis, e integração do frontend `useChat` com attachments.

Tudo que foi preparado nas stories anteriores (tools, schemas, UI shell) converge aqui.

**Specs de referência:**
- Tech Spec § 2 (instructions completo) + § 3 (working memory schema) + § 7 (streaming e rate limit): `docs/phase-09/tech-spec-arquiteto.md`
- Research § 1 (Mastra Memory config): `docs/phase-09/research-dependencies.md`

## Acceptance Criteria

### Architect Agent (packages/ai/src/mastra/agents/architect.ts)

1. GIVEN Agent criado com `id: 'architect'`, WHEN registrado na instância Mastra, THEN aparece no `mastra.getAgent('architect')`.
2. GIVEN Memory config, WHEN inicializado, THEN usa `PostgresStore` + `PgVector` (reuso 07A `DATABASE_URL`), `embedder: ModelRouterEmbeddingModel('openai/text-embedding-3-small')`, `options.lastMessages: 20`, `options.semanticRecall: { topK: 5, messageRange: { before: 2, after: 1 }, scope: 'resource', indexConfig: { type: 'hnsw', metric: 'dotproduct', m: 16, efConstruction: 64 } }`, `options.workingMemory: { enabled: true, schema: architectWorkingMemorySchema }`.
3. GIVEN model, WHEN config, THEN `model: 'openai/gpt-4o'` (forte, tech spec § 1).
4. GIVEN tools, WHEN config, THEN recebe `architectTools` (da story 08A.3).

### Working memory schema (packages/ai/src/mastra/types/architect-working-memory.ts)

5. GIVEN arquivo criado, WHEN exporta, THEN `architectWorkingMemorySchema` (Zod) matching exato tech spec § 3.1.
6. GIVEN type TS derivado, WHEN exportado, THEN `ArchitectWorkingMemory = z.infer<typeof architectWorkingMemorySchema>`.

### Instructions template (packages/ai/src/mastra/instructions/architect.ts)

7. GIVEN `buildArchitectInstructions(context: ArchitectContext): string`, WHEN chamado com `context = { templateId, currentStage, checklist, uploadedDocuments }`, THEN retorna string com todo o system prompt tech spec § 2.2 + template injection do vertical.
8. GIVEN 7 templates em `packages/ai/src/mastra/templates/` (clinical, ecommerce, real_estate, info_product, saas, local_services, custom), WHEN cada arquivo exporta `{ id, label, industry, emoji, promptInjection }`, THEN template específico adiciona perguntas-chave + presets recomendados (ex: clinical tem Rapport + Objeção).

### Route handler (apps/web/app/api/architect/chat/route.ts)

9. GIVEN POST `/api/architect/chat` com body `{ sessionId, messages, attachmentIds }`, WHEN user autenticado, THEN valida auth + rate limit + ownership da sessão.
10. GIVEN rate limit 10/min por session via Redis (tech spec § 7.3), WHEN exceeded, THEN retorna 429 com `retryAfter`.
11. GIVEN auth + rate OK, WHEN invoca, THEN: `mastra.getAgent('architect').stream(lastMsg.content, { memory: { thread: sessionId, resource: userId }, runtimeContext: { sessionId, userId, organizationId, attachmentIds }})`.
12. GIVEN stream retorna, WHEN response, THEN `result.toAIStreamResponse()` (Vercel AI SDK).

### Session persistence + auto-save (tech spec § 8)

13. GIVEN tool call do Arquiteto, WHEN roda, THEN Mastra persiste working memory automático em tabelas mastra_*.
14. GIVEN cada mensagem nova do usuário, WHEN chega, THEN atualiza `agent_creation_session.draftSnapshot` (cache denormalizado) via server action.
15. GIVEN rate limit ou erro, WHEN ocorre, THEN NÃO atualiza draftSnapshot com mensagem inválida.

### Frontend integration (useChat hook)

16. GIVEN hook `useArchitectChat(sessionId)` em `apps/web/modules/saas/agents/architect/hooks/useArchitectChat.ts`, WHEN chama `useChat({ api: '/api/architect/chat', id: sessionId, body: { sessionId }})`, THEN funciona com stream token-by-token.
17. GIVEN `handleSubmit` recebe attachments pending, WHEN chama, THEN envia no body `{ sessionId, attachmentIds: attachments.map(a => a.documentId) }`.
18. GIVEN `isLoading` true, WHEN compose, THEN `canSend: false`, botão enviar disabled.
19. GIVEN user aperta ESC, WHEN stream ativo, THEN `stop()` aborta mid-stream.

### Retomada de sessão

20. GIVEN `?session={id}` válido, WHEN rota carrega, THEN `useChat({ id: sessionId })` carrega histórico automaticamente via Mastra.
21. GIVEN primeira mensagem após retomada, WHEN Arquiteto responde, THEN usa working memory existente pra contextualizar ("Olá! Estávamos criando a Sofia...").

### StatusBar dinâmica

22. GIVEN working memory muda etapa (via tool call do Arquiteto), WHEN Realtime emite evento, THEN cliente recebe + StatusBar (09.2) atualiza check verde.

### Translations + error states

23. GIVEN erro 429 rate limit, WHEN catch, THEN toast `architect.errors.rateLimit` com countdown.
24. GIVEN erro de stream (network), WHEN catch, THEN retry automático (AI SDK built-in).

## Dev Agent Record

**Agent Model Used:** Neo (@dev) — caveman mode

**Completion Notes:**
- Memory config: `vector: getPgVector()` + `embedder: "openai/text-embedding-3-small"` + `semanticRecall` com `indexConfig.hnsw.{m, efConstruction}` (API v1.25 mudou shape: HNSW params ficam aninhados em `hnsw: {}`, não flat).
- `streamProtocol: 'text'`: MastraModelOutput em v1.25 expõe `textStream: ReadableStream<string>`, não tem `toAIStreamResponse()`. Usei plain `Response(textStream, { text/plain })` + useChat com `streamProtocol: 'text'`. AI SDK v4 consumer trata cada chunk como token.
- Placeholder `workingMemory` no `requestContext`: tools `architectTools` chamam `requireArchitectContext` que exige `workingMemory` presente. Como fonte de verdade fica no Mastra Memory (tabelas `mastra_*`), passo snapshot zerado derivado do `templateId`+`currentStage` só pra destravar. Próxima iteração pode hidratar via `memory.query(threadId).getWorkingMemory()`.
- ESC aborta stream via `onStop` no composer quando `isStreaming`.
- `ai/react` re-exporta `@ai-sdk/react` useChat — usei `ai/react` pra não precisar adicionar dep em apps/web.
- Retomada de sessão: `useChat({ id: sessionId })` carrega histórico automaticamente via AI SDK. `?session=xxx` já funcionava via 09.2.
- StatusBar dinâmica: Realtime subscribe em `agent_creation_session` via `useSessionEvents`. Quando Arquiteto avança etapa, tool call atualiza `draftSnapshot.currentStage` → Realtime emite → hook marca etapa anterior como `done` e troca `currentStage` atual.
- Rate limit config nova `ARCHITECT_CHAT_LIMIT` (10/60s por sessionId). Key inclui sessionId no slot do userId pra permitir múltiplas sessões paralelas do mesmo usuário.

**Divergências da spec:**
- AC22 (StatusBar via evento): implementado via Realtime na tabela `agent_creation_session` em vez de evento custom. Mesmo efeito, menos infra.
- AC24 (retry automático AI SDK built-in): `useChat` não tem retry built-in em v1.2. Deixei como TODO futuro — erro mostra toast + botão reload manual.
