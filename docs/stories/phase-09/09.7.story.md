# Story 09.7 — ArtifactInlineRefinement (Perfil do Negócio + Base de Conhecimento)

**Status:** Draft
**Phase:** 09
**Owner agent:** `@dev` (Neo)
**Story type:** UI component + form validation
**Estimated effort:** 2-3h

## Context

Pattern híbrido de refino (aprovado pelo Vinni, UI Spec Dilema 1-C): artefatos com 3-5 campos abrem inline expansion (card cresce em altura com formulário editável). Artefatos maiores (Blueprint) usam Dialog centralizado (09.8).

Esta story entrega o Inline Refinement pros 2 artefatos simples: Perfil do Negócio (5 campos) e Base de Conhecimento (3 campos).

**Specs de referência:**
- UI Spec § 6.2 (inline expansion wireframe): `docs/phase-09/ui-spec-arquiteto.md`
- UI Spec § 5.5.1 + § 5.5.3 (campos dos artefatos)

## Acceptance Criteria

### Trigger da inline expansion

1. GIVEN `ArtifactCard` artefato Perfil ou Conhecimento, WHEN user clica "Refinar artefato", THEN parent decide pattern (híbrido): para esses 2 tipos abre inline expansion (não Dialog).
2. GIVEN inline expansion ativo, WHEN renderiza, THEN card cresce em altura suave (`transition-all duration-300`), substitui conteúdo readonly por formulário editável.
3. GIVEN outro card inline aberto, WHEN user tenta abrir mais um, THEN fecha o anterior primeiro (um por vez).

### Form fields — Perfil do Negócio (5 campos)

4. GIVEN formulário Perfil, WHEN renderiza, THEN 5 campos:
   - Nome do negócio: `Input` obrigatório, max 80 chars
   - Resumo executivo: `Textarea` 3 rows, max 400 chars com counter
   - Serviços/produtos: `TagList` (reuso 07B v1) array de strings
   - Público-alvo: `Textarea` 2 rows, max 200 chars com counter
   - Objetivo do agente: `Textarea` 2 rows, max 300 chars com counter

### Form fields — Base de Conhecimento (3 campos)

5. GIVEN formulário Conhecimento, WHEN renderiza, THEN 3 campos:
   - Lista de documentos: componente read-only listando docs com botão `[X]` remover (se remove, marca pra remover sessão)
   - Upload novo documento: reuso `AttachmentMenu` da 09.4 (mini-card dentro do form)
   - Notas adicionais: `Textarea` 4 rows, max 1000 chars

### Actions

6. GIVEN form dirty, WHEN user clica "Cancelar", THEN volta ao estado readonly sem salvar, ignora mudanças.
7. GIVEN form válido, WHEN user clica "Salvar alterações", THEN: chama server action `updateArtifactInline(artifactId, data)` que chama `refineArtifact` tool indireto, aguarda response.
8. GIVEN save succeeds, WHEN retorna, THEN card colapsa pra readonly com conteúdo atualizado, Arquiteto recebe evento e envia mensagem de confirmação ("anotei, mudei o resumo").
9. GIVEN save fails, WHEN erro, THEN toast destructive + form mantém dirty pra retry.

### Validation

10. GIVEN form com campos obrigatórios vazios, WHEN submit, THEN zod valida, mostra errors inline, botão Salvar disabled.
11. GIVEN char limit atingido, WHEN user digita além, THEN textarea previne input adicional + counter destaque.

### Accessibility

12. GIVEN form, WHEN screen reader, THEN cada campo tem `<FormLabel>` visível + `aria-describedby` pra helper text.
13. GIVEN save/cancel buttons, WHEN focus, THEN focus ring visível.

### Integration

14. GIVEN parent (`ArtifactCard`), WHEN prop `pattern: 'inline'`, THEN renderiza `ArtifactInlineRefinement` embedded.
15. GIVEN server action chama tool `refineArtifact`, WHEN esta retorna erro `CONCURRENT_UPDATE` (race), THEN toast + refetch do artefato e user re-tenta.

## Scope IN

- Inline expansion com transição suave
- Forms pros 2 artefatos (Perfil + Conhecimento)
- Validação zod
- Server action `updateArtifactInline`
- Integração com TagList (reuso 07B)
- Upload de novo doc dentro do form Conhecimento
- Remoção de doc da sessão

## Scope OUT

- Blueprint form (09.8, em Dialog)
- Final Summary (readonly, sem refino)
- Drag-reorder de items em TagList (futuro 07C)
